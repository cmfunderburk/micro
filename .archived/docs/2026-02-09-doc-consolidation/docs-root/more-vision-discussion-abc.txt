  ---
  Key distinction I think matters

  There are two kinds of revelation:

  1. Natural revelation вҖ” You can see someone's holdings when you trade with them. This is "physics" вҖ” unavoidable given the interaction.
  2. Institutional revelation вҖ” You must disclose your alpha when proposing. This is a rule вҖ” a design choice that could be otherwise.

  This suggests a possible answer: natural revelation belongs to the environment (it's how the world works), while institutional revelation belongs to the market/matching institution (it's a rule humans impose).

  ---
  Concrete examples to test intuitions

  Scenario A: Job market
  - Proposer (firm) reveals: wage offer, job description
  - Target (worker) reveals: nothing until acceptance (then credentials verified)
  - Asymmetry: proposer reveals more

  Scenario B: Housing market with disclosure laws
  - Proposer (buyer) reveals: offer price, financing status
  - Target (seller) reveals: defects, liens, inspection reports (mandatory)
  - Asymmetry: target reveals more (due to regulation)

  Scenario C: Anonymous exchange
  - Neither reveals anything until trade executes
  - Then both observe the actual allocation

  These feel like market design choices, not environmental facts. That pushes toward institution-level.

  ---
  A hybrid possibility

  Split the current InformationEnvironment concept:

  - ObservationEnvironment вҖ” Ambient, from-a-distance observation (what you can see without interacting). Current FullInformation / NoisyAlphaInformation live here.
  - RevelationRules вҖ” Interaction-triggered revelation. Attached to matching institution or as a parameter of it.

  This keeps environmental facts separate from institutional rules while avoiding a fully separate abstraction.

  ---
  The Timing Problem

  Within the Execute phase, there's a sequencing question:

  1. Proposals submitted
  2. ??? Revelation happens ???
  3. Targets evaluate and accept/reject
  4. Trades execute

  For Model C to work, revelation must happen before accept/reject decisions. The target needs the revealed information to compute posterior expected surplus.

  This suggests Execute has sub-phases:

  Execute:
    a. Collect all proposals
    b. Apply revelation_rules.on_proposal() for each
    c. Update beliefs for proposers and targets
    d. Targets evaluate proposals using updated beliefs
    e. Process acceptances/rejections
    f. Apply revelation_rules.on_rejection() for rejects
    g. Execute trades
    h. Apply revelation_rules.on_trade() for completed trades
    i. Final belief updates

  This is more complex than the current Execute, but it makes the information flow explicit.

  Revelation-as-Action-Property: Two Interpretations

  Interpretation 1: Revelation is intrinsic to action type

  Each action class defines what it reveals:

  class ProposeAction(Action):

          # Hardcoded: proposal reveals X to target, Y to proposer

  Problem: To compare different revelation rules, you'd need different action classes (ProposeWithFullRevelation, ProposeWithNoRevelation). Awkward.

  Interpretation 2: Actions generate revelation; rules interpret it

  Actions produce "revelation events" вҖ” raw facts about what happened. A configurable RevelationRules component determines what agents actually learn from those events.

  class ProposeAction(Action):
      def revelation_event(self) -> RevelationEvent:
          return ProposalEvent(proposer=self.agent, target=self.target)

  # Separate, configurable:
  class RevelationRules:
      def interpret(self, event: RevelationEvent) -> AgentLearnings:
          # Given this event occurred, what do participants learn?

  This preserves both:
  - Mental model: revelation flows from actions
  - Configurability: same actions, different rules for what's learned
