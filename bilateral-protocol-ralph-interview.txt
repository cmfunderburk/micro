  ---
  Protocol Implementation Discussion

  Based on the roadmap, here's my understanding of each:

  1. TIOLI (Take-It-Or-Leave-It)

  Mechanism: Proposer makes single offer at responder's indifference point (extracts maximum surplus consistent with participation).

  Implementation: Straightforward - it's essentially "Nash bargaining with ОІ=1 for proposer". The offer gives responder exactly their disagreement utility.

  Open design choice (marked configurable): Proposer selection rule
  - Random (coin flip)
  - First-mover (who initiated meeting)
  - By type (e.g., higher Оұ proposes)
  - Fixed role per agent

  My suggestion: Make this a protocol parameter, default to random. Could later tie to the tick model's "higher surplus proposes" rule.

  ---
  2. Kalai-Smorodinsky

  Mechanism: Find the point on the Pareto frontier where:
  (u_A - d_A) / (max_A - d_A) = (u_B - d_B) / (max_B - d_B)
  where max_A is A's maximum achievable utility (when B gets only their disagreement payoff).

  Implementation: Requires computing the "utopia point" (max achievable for each agent), then finding where the line from disagreement to utopia intersects the Pareto frontier. Numerically similar to Nash - golden section search along contract curve.

  Key difference from Nash: More sensitive to the shape of the feasible set. In symmetric cases, identical to Nash.

  ---
  3. Asymmetric Nash

  Mechanism: Maximize (u_A - d_A)^ОІ Г— (u_B - d_B)^(1-ОІ) where ОІ is A's bargaining power.

  Implementation: Nearly identical to current Nash - just weight the exponents differently in the optimization.

  Design decision (from roadmap): ОІ derived from per-agent weights: ОІ = w_A / (w_A + w_B). This is elegant - agents have an intrinsic "bargaining power" attribute.

  Question: Where does w live? Options:
  - On AgentPrivateState (intrinsic, hidden)
  - On AgentType (observable)
  - As protocol parameter (scenario-level)

  ---
  Implementation Questions for You

  1. For TIOLI proposer selection: Start with random, or implement multiple modes from day one?
  2. For Asymmetric Nash weights: Should w be a new agent attribute, or should we derive it from discount_factor (connecting to Rubinstein's patience-as-power interpretation)?
  3. Test strategy: The existing theory tests validate Nash/Rubinstein against textbook properties. Similar tests for these three? (K-S monotonicity, Asymmetric Nash reducing to symmetric at ОІ=0.5, TIOLI giving proposer full surplus)

  Phase 3: Verification Design (Research Software)

  Since this is research software, theory tests are primary.

  TIOLI Protocol

  Canonical source: O&R Bargaining Ch 3

  Testable properties:
  1. Proposer extracts full surplus: Proposer's utility = maximum achievable given responder's IR constraint
  2. Responder at indifference: Responder utility = disagreement utility (within tolerance)
  3. Pareto efficiency: Outcome on contract curve
  4. Proposer identity matters: Swapping proposer swaps who gets surplus

  Tolerance: 1e-6 (closed-form solution)

  Asymmetric Nash Protocol (Power-weighted)

  Canonical source: O&R Bargaining Ch 2

  Testable properties:
  1. Reduces to symmetric Nash: When w1 = w2, outcome equals NashBargainingProtocol

  4. Individual rationality: Both agents at least as well off as disagreement

  Tolerance: 1e-6 (uses existing solver)

  Note: Some tests already exist in test_nash_bargaining.py::TestAsymmetricNash for the underlying function. New tests focus on the protocol wrapper and bargaining_power attribute integration.
