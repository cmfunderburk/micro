  Cons:
  - Loses individual commit history (though branch can be kept)
  - Large diff in one commit (~35k lines)
  - Loses attribution granularity

  ---
  Option 2: Merge Commit (preserve history)
  git checkout main
  git merge tweaking

  Pros:
  - Preserves full development history
  - Each change traceable to its commit
  - Standard open-source workflow

  Cons:
  - 82 commits of varying quality/granularity
  - History includes WIP commits, fixups, etc.
  - Harder to understand "what changed in 0.2.0"

  ---
  Option 3: Interactive Rebase + Merge (curated history)
  git rebase -i main  # Squash related commits
  git checkout main
  git merge tweaking

  Pros:
  - Curated commit history (maybe 10-15 logical commits)
  - Preserves meaningful structure
  - Best of both worlds

  Cons:
  - Time-consuming to curate 82 commits
  - Risk of mistakes during rebase
