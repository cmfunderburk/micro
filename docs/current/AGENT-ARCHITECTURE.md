# Agent Architecture

**Status:** Source of truth (implementation-aligned)  
**Last Updated:** 2026-02-09

## 1. Objective
This document defines the active agent architecture and simulation semantics used by the current codebase.

It is intended to answer:
- what an agent is in this platform
- how a tick executes
- where institutions plug in
- which behaviors are normative vs currently deferred

## 3. Agent Model

## 3.1 State Model

```text
Agent
├── private_state: AgentPrivateState
│   ├── preferences: CobbDouglas(alpha)
│   └── endowment: Bundle (immutable disagreement reference)
│
├── holdings: Bundle (mutable current inventory)
│
├── interaction_state: AgentInteractionState
│   ├── state: AVAILABLE | PROPOSAL_PENDING | NEGOTIATING
│   ├── proposal_target / proposal_tick
│   ├── negotiation_partner / negotiation_start_tick
│   └── cooldowns: Dict[agent_id, ticks]
│
├── decision attributes
│   ├── perception_radius
│   ├── discount_factor
│   ├── bargaining_power
│   ├── movement_budget
│   └── opportunity_cost
│
└── beliefs (optional)
    ├── memory
    ├── price_belief
    ├── type_beliefs
    └── update_rule
```

## 3.2 Key Invariants
1. `endowment` is immutable post-construction.
2. `holdings` is the state updated by trade.
3. `opportunity_cost` is set during Decide and consumed during Execute acceptance checks.
4. Cooldowns are unilateral proposer constraints, not shared commitments.
5. Belief subsystem must remain optional (agents run without beliefs enabled).

## 4. Action Model

## 4.1 Action Types
Decide-phase selectable actions:
- `MoveAction`
- `ProposeAction`
- `WaitAction`

Reserved action classes (not currently selected in Decide):
- `AcceptAction`
- `RejectAction`

Current acceptance/rejection behavior is institutionalized in Execute through `evaluate_proposal`, not explicit action submission.

## 4.2 Fallback Rule
`ProposeAction` carries a fallback action.
- On explicit rejection: cooldown is applied and fallback executes.
- On implicit non-selection: fallback executes without cooldown.

This prevents proposal failure from producing undefined or dead action paths.

## 5. Tick Semantics (Current Implementation)

Each tick executes:
1. **Pre-tick maintenance**: cooldown decrement.
2. **Perceive**: evaluate visible targets under current information environment.
3. **Decide**: each agent selects one action.
4. **Execute**:
- resolve mutual proposals
- resolve non-mutual proposals via immediate accept/reject evaluation
- execute fallbacks for failed proposals
- execute movement for remaining eligible agents

## 5.1 Current Execution Characteristics
- Proposal resolution is same-tick (practically no multi-tick pending negotiation flow).
- Mutual proposals are processed prior to non-mutual proposal handling.
- Accepted proposals execute bargaining and apply holdings updates in same tick.

## 6. Institutional Components

## 6.1 InformationEnvironment
Defines what is observable:
- `FullInformation`
- `NoisyAlphaInformation`

Used in perception and target evaluation paths.

## 6.2 DecisionProcedure
Defines how agents choose and evaluate actions.
Current default:
- `RationalDecisionProcedure`

Acceptance predicate in current default:
- accept proposal iff expected surplus `>=` own stored `opportunity_cost`

## 6.3 BargainingProtocol
Defines bilateral allocation outcomes for matched agent pairs.
Implemented protocols:
- `NashBargainingProtocol`
- `RubinsteinBargainingProtocol` (BRW mapping)
- `AsymmetricNashBargainingProtocol`
- `TIOLIBargainingProtocol`

## 6.4 Matching/Clearing Status
There is no independent `MatchingProtocol` runtime abstraction currently.
Matching/clearing behavior is embedded in proposal resolution logic within `Simulation._execute_actions`.

This is functional but is the major remaining modularity gap relative to the vision.

## 7. Belief Architecture

## 7.1 Implemented
- Agent memory (trades, interactions, observed prices)
- Type beliefs (believed alpha + confidence + interaction count)
- Price belief (mean/variance/observation count)
- Bayesian and heuristic update rules
- Trade-time belief updates
- Optional use of beliefs in search/evaluation

## 7.2 Not Fully Causal Yet
- Price beliefs are weakly integrated in behavior.
- Belief-driven acceptance/posterior-driven revelation model is not the active runtime default.

## 8. Data and Logging Contracts

## 8.1 Runtime state contract
Primary runtime fields exposed to backend/frontend:
- tick
- agents (id, position, holdings, alpha, utility, interaction state, etc.)
- trades (tick-level plus cumulative history)
- metrics (welfare, gains, cumulative trades)
- beliefs (if enabled)

## 8.2 Log contract
Structured per-tick logs include:
- agent snapshots
- search decisions and target evaluations
- movement events
- trade events
- optional belief snapshots

## 9. Integration Surface

## 9.1 Backend
`server/simulation_manager.py` maps simulation state to API/WebSocket payloads.

## 9.2 Frontend
`frontend/src/types/simulation.ts` defines client contracts consumed by Zustand stores and mode-specific UIs.
