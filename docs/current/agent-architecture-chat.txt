  My Proposed Agent Identity Statement

  Let me propose a foundational definition, then ask for your input:

  An agent is an economic decision-maker characterized by:
  1. Preferences over bundles of goods (utility function)
  2. Holdings (current endowment, position)
  3. Beliefs about the world (other agents, prices, opportunities)
  4. Action capacity constrained by time and space
  5. Decision rule mapping perceived state to action choice

  The agent's objective is utility maximization, but the decision rule may be:
  - Fully rational (compute optimal action)
  - Boundedly rational (satisfice, use heuristics)
  - Adaptive (learn from experience)

  This is an experimental variable, not a fixed assumption.

  Questions for You (Phase 0)

  Q0.1: Agent as decision-maker vs agent as state-holder

  The current implementation treats agents as primarily state containers - they hold preferences and endowments, but protocols "do things to them" (bargaining protocols compute outcomes, search modules evaluate targets).

  The roadmap moves toward agents as autonomous decision-makers who consent to trades.

  Where on this spectrum should the agent model sit?
  - (a) State-centric: Agent is a bundle of attributes; protocols/institutions operate on agents
  - (b) Decision-centric: Agent has a decision procedure; institutions constrain the decision space
  - (c) Hybrid: Agent state is passive; agent decision rule is active; both are first-class

  My recommendation: (c) Hybrid - this matches the theoretical tradition where preferences are stable but behavior is the outcome of optimization/choice.

  Q0.2: The status of "type" in the agent model

  Currently AgentType is "what others observe about me." But the roadmap's action-budget model has agents choosing actions. Should type include:
  - (a) Only intrinsic attributes (preferences, discount factor, bargaining power)
  - (b) Intrinsic + observable state (includes current holdings, position)
  - (c) Intrinsic + observable state + behavioral tendencies (includes decision rule class)

  The theoretical question: when I observe another agent, what information might I extract?

  Q0.3: The role of "self-knowledge"

  Does an agent know its own type perfectly? Standard micro assumes yes (I know my preferences). But we could model:
  - Self-uncertainty about preferences (behavioral economics)
  - Learning about own production function

  For now, I assume: Agents have perfect self-knowledge of preferences and endowments. Confirm or adjust?

  Phase 0 Resolution

  Agent Identity (Revised)

  An agent is an autonomous economic decision-maker characterized by:
  1. Attributes: Preferences, holdings, patience, bargaining power, spatial position
  2. Beliefs: What the agent thinks about the world (other agents, prices, opportunities)

  On "Type" (Key Insight)

  The game-theoretic concept of type (private information determining payoffs) should not be reified as a class within Agent. Instead:

  - Agent has attributes (preferences, endowment, discount_factor, bargaining_power, etc.)
  - Information environment determines which attributes are observable to whom
  - Game-theoretic type is derived when needed for theoretical analysis, not stored

  This cleanly separates:
  - What an agent is (attributes)
  - What others can see (observation function)
  - What a theorist might model (type in mechanism design sense)

  1.2 Questions for You (Phase 1)

  Q1.1: Preference representation

  Currently we have CobbDouglas(alpha) only. The theoretical foundations list several functional forms:
  - Cobb-Douglas: u(x,y) = x^Оұ y^(1-Оұ)
  - CES: u(x,y) = (Оұx^ПҒ + (1-Оұ)y^ПҒ)^(1/ПҒ)
  - Linear: u(x,y) = Оұx + ОІy
  - Leontief: u(x,y) = min(Оұx, ОІy)
  - Quasi-linear: u(x,y) = x + v(y)

  Should the agent architecture:
  - (a) Assume a common functional form (all agents Cobb-Douglas, vary Оұ only)
  - (b) Allow heterogeneous functional forms (agent A is Cobb-Douglas, agent B is CES)
  - (c) Abstract over functional form (Preferences interface with utility(), MRS(), demand() methods)

  My recommendation: (c) for the architecture document, even if initial implementation is (a). This future-proofs for studying how preference structure affects market outcomes.

  Q1.2: What is "endowment" vs "holdings"?

  Theoretical distinction:
  - Endowment: Initial allocation at start of simulation
  - Holdings: Current bundle after trades

  Should we track both, or just current holdings?
  - (a) Track both (enables computing gains from trade relative to initial state)
  - (b) Holdings only (simpler; initial state recoverable from logs)

  My lean: (a) - the initial endowment is theoretically meaningful (it's the disagreement point in Nash bargaining, the status quo for welfare comparisons).

  Q1.3: Time preferences and bargaining power

  These are currently separate:
  - discount_factor (Оҙ): Used in Rubinstein bargaining, represents patience
  - bargaining_power (w): Used in Asymmetric Nash, represents exogenous power

  In BRW (1986), asymmetric Nash weights derive from discount factors as ОҙвҶ’1. Should we:
  - (a) Keep them separate (different theoretical constructs, configured independently)
  - (b) Derive bargaining power from discount factor (implement BRW relationship)
  - (c) Make it configurable (can be independent or derived)

  Theoretical note: They measure different things. Discount factor is about impatience (time cost of delay). Bargaining power is about leverage (ability to capture surplus). These can diverge if power comes from sources other than patience (institutional position, outside options, commitment ability).

  Q1.4: Should "action state" be part of the agent?

  The roadmap introduces:
  - Rejection cooldowns (can't re-engage rejected partner for K ticks)
  - Possibly: action history, fatigue, learning state

  Should these be:
  - (a) Agent attributes (agent carries this state)
  - (b) Simulation-level tracking (simulation tracks which pairs are on cooldown)
  - (c) Hybrid (agent tracks own state; simulation tracks interaction-pair state)

  ---
  Q1.5: Beliefs architecture depth

  Current implementation has:
  - Memory (trade history, observations)
  - Type beliefs (what I think about others' preferences)
  - Price beliefs (expected exchange rates)

  The roadmap suggests beliefs are deferred from decision logic. How deeply should beliefs be specified in the agent architecture?
  - (a) Minimal: Beliefs exist as an optional component; specifics left to implementation
  - (b) Structured: Define belief categories (about others, about prices, about environment) but not update rules
  - (c) Full specification: Define belief structure, update interfaces, and how beliefs feed into decisions

  2.3 How Do Beliefs Update?

  This is where theory meets implementation difficulty.

  Bayesian updating (theoretically canonical):
  - Prior: P(Оё)
  - Observation: signal s with likelihood P(s|Оё)
  - Posterior: P(Оё|s) вҲқ P(s|Оё)P(Оё)

  Challenges in our context:
  1. What's the likelihood function? If I observe a trade at price p, what does that tell me about Оё?
  2. Computational tractability: Full Bayesian updating over high-dimensional type spaces is expensive
  3. Common knowledge: Standard game theory assumes agents know others are Bayesian updaters

  Heuristic updating (bounded rationality):
  - Exponential moving average: belief вҶҗ (1-О»)В·belief + О»В·observation
  - Simpler, but ad-hoc
  - Needs theoretical justification (when is this "close enough" to Bayesian?)

  Current implementation uses both, with a swappable UpdateRule interface. But the user correctly notes: beliefs don't yet affect behavior.

  2.4 The Hard Problem: Beliefs вҶ’ Decisions

  Here's where it gets thorny. Consider this decision problem:

  Agent A is choosing whether to approach Agent B (who is distant) or Agent C (who is nearby).
  A has beliefs about B's and C's types.
  A uses discount factor Оҙ to value future payoffs.

  Theoretically correct approach:
  1. Compute expected surplus with B, given beliefs about B's type
  2. Discount by time to reach B (distance / movement_budget, discounted by Оҙ)
  3. Same for C
  4. Choose argmax

  The problems:

  Problem 1: What surplus function?
  Under Nash bargaining, surplus depends on both types. If I'm uncertain about your type, I should compute expected surplus:
  E[surplus(A, B)] = вҲ« surplus(A, Оё_B) В· P(Оё_B) dОё_B
  This requires integrating over belief distribution. Computationally expensive; also, the agent needs to "know" the surplus function (Nash, Rubinstein, etc.).

  Problem 2: Strategic considerations
  If A approaches B, B might reject. Should A's beliefs include P(B accepts | I approach)?
  This gets into game-theoretic reasoning about others' strategies, not just types.

  Problem 3: Consistency across agents
  If all agents are doing Bayesian reasoning about each other, we're in a Bayesian game. Equilibrium concepts (BNE, PBE) require beliefs to be consistent with strategies.

  In simulation, we can't guarantee thisвҖ”agents update based on observations, but observations depend on others' strategies, which depend on their beliefs...

  2.6 Questions for You (Phase 2)

  Q2.1: Point estimates vs full distributions

  Should decision-making use:
  - (a) Point estimates (expected type, then compute surplus with that type)
  - (b) Expected surplus (integrate surplus over type distribution)
  - (c) Configurable (point estimates as default; expected surplus for "sophisticated" agents)

  Trade-off: (a) is simpler and faster; (b) is more theoretically correct but computationally heavier.

  Q2.2: Do agents know the bargaining protocol?

  For expected_surplus(partner, protocol) to work, agents must "know" how surplus is computed. This is:
  - (a) Fine вҖ” institutional rules are common knowledge (standard game theory assumption)
  - (b) Problematic вҖ” agents should learn the protocol from experience
  - (c) Protocol-specific вҖ” for some protocols (Nash, TIOLI) rules are clear; for others (Nash Demand) agents don't know what partner will demand

  Q2.3: Acceptance probability

  Should beliefs include "will this partner accept my proposal?"
  - (a) No вҖ” assume all positive-surplus trades execute (current model)
  - (b) Yes, simple model вҖ” acceptance probability based on observed rejection rates
  - (c) Yes, strategic model вҖ” infer acceptance from beliefs about partner's outside options

  This connects to the consent model in the roadmap.

  Q2.4: Belief consistency across agents

  In standard Bayesian games, beliefs must be consistent (derived from common prior, updated by commonly known strategies). We can't guarantee this in simulation. Options:
  - (a) Accept inconsistency вҖ” agents have subjective beliefs, may be mutually inconsistent; this is realistic but hard to interpret theoretically
  - (b) Impose structure вҖ” all agents start from common prior, update by same rule; divergence is due to different observations only
  - (c) Study as variable вҖ” inconsistency itself is an experimental variable (how does belief heterogeneity affect outcomes?)

  My lean: (b) with (c) as extension. Common prior + common update rule is the standard theoretical assumption; relaxing it is interesting for research.
