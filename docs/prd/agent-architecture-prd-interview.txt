  Research Use Cases

  What do we need to measure for institutional economics research?

  1. Transaction costs вҖ” How many ticks did this exchange consume?
  2. Coordination failures вҖ” How many proposals were rejected? Why?
  3. Protocol comparison вҖ” Same economy, different institutions вҶ’ different tick costs?
  4. Exchange tracing вҖ” Follow a single exchange from proposal to outcome
  5. Welfare analysis вҖ” Who gained what, when?

  Architecture Options

  Option A: Event-Sourced

  Log every atomic action as an immutable fact:
  ProposalEvent(tick=5, proposer=A, target=B, exchange_id=X)
  AcceptanceEvent(tick=6, acceptor=B, proposer=A, exchange_id=X)
  NegotiationStartEvent(tick=6, pair=(A,B), protocol=Nash, exchange_id=X)
  TradeEvent(tick=7, pair=(A,B), bundles=..., exchange_id=X)

  - Pro: Complete audit trail, can reconstruct any state
  - Pro: Events are facts, analysis is queries over facts
  - Pro: exchange_id links the full sequence вҖ” can measure "proposal to completion = 2 ticks"
  - Con: More verbose
  - Con: Need to reconstruct "what was true at tick N" from event stream

  Option B: State-Snapshot

  Log full state at each tick:
  TickRecord(tick=5,
    agents=[{id=A, state=ProposalPending(B), holdings=...}, ...],
    events=[...])

  - Pro: Easy to see "what was true at tick N"
  - Pro: Good for visualization (render current state)
  - Con: Loses causation вҖ” why did state change?
  - Con: Hard to trace multi-tick exchanges
  - Con: Redundant data (full state every tick)

  Option C: Hybrid with Correlation

  Events as source of truth, state snapshots for convenience:
  TickRecord(
    tick=5,
    agent_states={A: ProposalPending(B), B: Available, ...},  # Snapshot
    events=[ProposalEvent(exchange_id=X, ...)],               # What happened
  )

  - Pro: Best of both вҖ” can query events OR inspect snapshots
  - Pro: exchange_id enables exchange-level analysis
  - Pro: State snapshot enables visualization without event replay
  - Con: Some redundancy (state derivable from events)

  My Recommendation

  Option C (Hybrid) with these specifics:

  1. Atomic event types вҖ” Each action is its own event class:
    - ProposalEvent(tick, proposer, target, exchange_id)
    - ProposalResponseEvent(tick, responder, proposer, response: Accept|Reject|Timeout|CoLocationLost, exchange_id)
    - NegotiationEvent(tick, pair, protocol, outcome: Success|Failure, exchange_id)
    - TradeEvent(tick, pair, bundles_exchanged, exchange_id) вҖ” already exists, add exchange_id
    - CooldownEvent(tick, agent, target, duration, reason)
  2. Correlation via exchange_id вҖ” UUID generated when proposal created, carried through entire sequence. Enables:
    - "Show me all events for exchange X"
    - "Average ticks from proposal to completion under Nash vs Rubinstein"
    - "What fraction of proposals result in trades?"
  3. State snapshot in TickRecord вҖ” Lightweight summary of interaction states for visualization:
  interaction_states: Dict[str, InteractionState]  # agent_id вҶ’ state
  4. TickRecord.events remains a list вҖ” All events that occurred this tick, queryable by type
